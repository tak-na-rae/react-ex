React.memo & useCallback

렌더링 최적화


리액트 컴포넌트가 렌더링 하는 내용은 다음 두 데이터에 의해 결정된다.

a) 컴포넌트가 생성한 리액트 상태 (useState)
b) 부모 컴포넌트로부터 전달 받은 props

따라서 위 두 값 중 하나가 변경되면,
컴포넌트는 다시 렌더링 되어야 한다.

부모 컴포넌트가 렌더링 될 때, 모든 자식 컴포넌트들도 렌더링 된다.

상태, props 둘 다 변경되지 않은 컴포넌트들은 다시 렌더링 되지 않게 만들기 -> 렌더링 최적화

CounterA.js

import React, { useState } from 'react'

function CounterA() {
	const [value, setValue] = useState(0);
	return (
		<div className="box">
			<h1>CounterA</h1>
			<button onClick={() => setValue(value + 1)}>
				{ value }
			</button>
		</div>
	)
}

export default CounterA;

버튼이 클릭되고 value 상태 값이 증가하면면 setValue(value + 1),
이 컴포넌트는 다시 렌더링 되어야 한다.

컴포넌트가 다시 렌더링 되어야 { value } 부분에 새 상태 값을 출력될 수 있다.

a) 어떤 기술로 만들었는지는 상관 없이, 컴포넌트가 사용하는 상태 데이터
b) 부모 컴포넌트로부터 전달 받은 props
위 두 값 중 하나가 변경되면, 컴포넌트는 다시 렌더링 되어야 한다.

기본 렌더링 정책

부모 컴포넌트가 다시 렌더링 될 때, 자식 컴포넌트들도 전부 다시 렌더링 된다.
이것이 기본 렌더링 정책이다.

그래서, 새 Todo 항목이 추가되어, TodoRoot 컴포넌트의 todoList 상태가 변경되면,
TodoList 아래 모든 컴포넌트들이 다시 렌더링 되어야 한다.

그런데 새 Todo 항목만 추가 되었을뿐이고,
todoList에 이미 들어있던 Todo 항목의 값이 변경되지 않았는데,
변경되지도 않은 Todo 항목들도 다시 렌더링 되어야하지?
이것은 낭비 아닌가?

렌더링 최적화

부모 컴포넌트가 다시 렌더링 될 때, 자식 컴포넌트들도 전부 다시 렌더링 되는 것이 기본 정책이지만,

a) 그 컴포넌트가 관리하는 리액트 상태
b) 부모 컴포넌트로부터 전달 받은 props

위 두 값이 변경되지 않은 자식 컴포넌튼 다시 렌더링 될 필요가 없다.

위 두 값이 변경되었는지 확인해서,
불필요한 렌더링은 막아주는 리액트의 기능이 React.memo 이다.

렌더링 최적화와 props


리액트 컴포넌트가 렌더링 하는 내용은 다음 두 데이터에 의해 결정된다.

a) 컴포넌트가 생성한 리액트 상태 (useState)
b) 부모 컴포넌트로부터 전달 받은 props

따라서 위 두 값 중 하나가 변경되면,
컴포넌트는 다시 렌더링 되어야 한다.

중요한 데이터와 그 데이터에 대한 작업은 부모 container 컴포넌트에 모아서 구현하고,
그렇게 구현된 데이터와 작업 함수를 presentation 컴포넌트에 props로 전달하는 것이 보통이다.

그렇게 구현된 경우에도 렌더링 최적화가 잘 작동하는지 살펴보자.

CounterContainer.js

import React, { useState } from 'react'
import CounterA from './CounterA'
import CounterB from './CounterB'

function CounterContainer() {
	console.log('CounterContainer 호출됨');
	const [countA, setCountA] = useState(0)
	const [countB, setCountB] = useState(0)
	const [value, setValue] = useState(0);

	const increaseA = () => { setCountA(countA + 1); }
	const increaseB = () => { setCountB(countB + 1); }
	
	return (
		<div className="box">
			<h1>CounterContainer</h1>
			<button onClick={() => setValue(value + 1)}>
				{ value }
			</button>
			<CounterA value={countA} increase={increaseA} />
			<CounterB value={countB} increase={increaseB} />
		</div>
	)
}

export default React.memo(CounterContainer);

버튼을 클릭할 때 마다,
CounterContainer, CounterA, CounterB 컴포넌트들이 언제나 다시 렌더링된다는 것을
chrome 개발자 창의 console 탭에서 확인할 수 있다.

그 이유는 다음과 같다.

function CounterContainer() {
	console.log('CounterContainer 호출됨');
	const [countA, setCountA] = useState(0)
	const [countB, setCountB] = useState(0)
	const [value, setValue] = useState(0);
	
	const increaseA = () => { setCountA(countA + 1); }
	const increaseB = () => { setCountB(countB + 1); }

return (
		<div className='box'>
			<h1>CounterContainer</h1>
			<button onClick={() => setValue(value + 1)}>
				{ value }
			</button>
			<CounterA value={countA} increase={increaseA} />
			<CounterB value={countB} increase={increaseB} />
		</div>
		)
}

increaseA, increaseB 함수가 매번 새로 만들어져서
각각 CounterA, CounterB 컴포넌트의 props로 전달되기 때문이다.

전달된 props 값이 이전과 다르면 컴포넌트는 다시 렌더링 된다.

  1. CounterA, CounterB 둘 중 하나의 버튼이 클릭되면
  1. CounterContainer 컴포넌트에서 만들어 전달한 increseA, increaseB 둘 중의 한 함수가 호출되고
  1. setCountA, setCountB 둘 중의 하나가 호출된다.
    즉 CounterContainer 컴포넌트의 상태값이 변경된다.
  1. CounterContainer 컴포넌트가 다시 렌더링된다.
    즉 CounterCountainer 함수가 호출된다.
  1. CounterContainer 함수 본문에서 increaseA, increaseB의 inner 함수가 다시 생성된다.
  1. 그렇게 새로 생성된 increaseA, increaseB 함수가
    각각 CounterA, CounterB 컴포넌트의 props로 전달된다.
  1. 비록 value props 값은 전과 같더라도
    props로 전달된 increase 함수가 전과 다른 함수이기 때문에,
    CounterA, CounterB 컴포넌트는 둘 다 다시 렌더링 된다.

useCallback 훅

javascript의 closure

closure1.js

function outterFunc(a) {
	let innerFunc = (i) => i * a;
	return innerFunc;
}

const f1 = outterFunc(2);
const f2 = outterFunc(4);

console.log(f1(3)); //6
console.log(f2(3)); //12

outterFunc 함수는 innerFunc 함수를 만들어 리턴한다.

let innerFunc = (i) => i * a;

이 코드가 실행될 때 마다, 새 함수가 생성되어 innerFunc 지역 변수에 대입된다.
이렇게 만들어지는 함수의 본문에서 a 변수를 사용하고 있다.
a 지역 변수는 outterFunnc 함수의 변수이다.

inner 함수 본문에서 outter 함수의 변수를 사용하다면,
inner 함수가 만들어지는 시점의 outter 함수의 변수가 inner 함수 내부에 캡처 된다.
이렇게 outter 함수의 변수를 캡처한 inner 함수를 closure라고도 부른다.

어떤 함수의 지역 변수와 파라미터 변수는, 그 함수가 호출될 때 만들어지고, 리턴할 때 사라진다.
즉 outterFunc 함수의 변수 a는, outterFunc 함수가 호출될 때마다 다시 만들어진다.

const f1 = outterFunc(2);

이렇게 outterFunc 함수가 호출될 때 만들어진 a 변수의 값은 2 이다.
f1 변수에 대입되는 함수는 (i) => i * 2 이다.

const f2 = outterFunc(4);

이렇게 outterFunc 함수가 호출될 때 만들어진 a 변수의 값은 4 이다.
f2 변수에 대입되는 함수는 (i) => i * 4 이다.

outterFunc 함수가 두 번 호출되었으니,

이 함수의 변수 a도 두 번 만들어졌고,
이 함수 내부의 inner 함수도 두 번 만들어졌다.

이렇게 만들어진 두 개의 a 변수는 서로 다른 변수이고,
이렇게 만들어진 두 개의 inner 함수도 서로 다른 함수이다.

App.js

import React, { useState } from 'react';

function App() {
	const [count, setCount] = useState(0);
	const increase = () => setCount(count + 1);
	return (
		<div>
			<h1>{ count }</h1>
			<button onClick={increase}>increase</button>
		</div>
	);
}

export default App;

App 컴포넌트가 랜더링 될 때 마다, App 함수가 호출된다.
App 함수가 호출될 때 마다 하늘색으로 칠한 inner 함수가 다시 생성된다.

하늘색 inner 함수 본문의 count는 outer 함수에 선언된 지역 변수이다.
이 count 값이 inner 함수 본문에 캡쳐되어 포함된다.
즉 하늘색 inner 함수가 생성될 때 마다, inner 함수 본문에, 현재 count 값이 캡쳐되어 포함된다.

버튼이 클릭될 때마다 count 값이 변하기 때문에,
그 새 count 값을 캡쳐하여 하늘색 inner 함수가 새로 만들어질 수 밖에 없다.

() => setCount(count + 1)
외부의 count 값을 캡쳐하는 하늘색 inner 함수는 순수한 함수가 아니다.
이 count 값이 변경될 때 마다 하늘색 inner 함수가 새로 만들어져야 한다.

다음과 같이 구현할 수도 있다.

import React, { useState } from 'react';

function App() {
	const [count, setCount] = useState(0);
	const increase = () => setCount(count => count + 1);
	return (
		<div>
			<h1>{ count }</h1>
			<button onClick={increase}>increase</button>
		</div>
	);
}

export default App;

() => setCount(count => count + 1)

이 inner 함수는 outter 함수의 count 값을 캡쳐하지 않는다.

이 inner 함수 본문에서 setCount 함수의 파라미터값은 count => count + 1 함수이다.
이 함수는 setCount 함수 내부에서 호출된다.
이 함수가 setCount 함수 내부에서 호출될 때, 현재 count 값이 count 파라미터 변수에 전달된다.

count은 App 함수의 지역 변수이다.
count는 count => count + 1 함수의 파라미터 변수이다.

() => setCount(count => count + 1)

이 연두색 inner 함수는 count 값을 캡쳐하지 않는다.
따라서 이 연두색 inner 함수는 count 값이 변경될 때 마다 다시 만들 필요 없다.

그렇지만 위와 같이 평범한 inner 함수로 구현하면
App 함수가 호출될 때 마다 연두색 inner 함수가 매번 새로 만들어진다.
이렇게 inner 함수가 불필요하게 새로 만들어지는 것을 막으려면 useCallback 훅을 사용한다.

useCallback 훅

import React, { useState, useCallback } from 'react';

function App() {
	const [count, setCount] = useState(0);
	const increase = useCallback(() => setCount(count => count + 1), [setCount]);
	return (
		<div>
			<h1>{ count }</h1>
			<button onClick={increase}>increase</button>
		</div>
	);
}

export default App;

매번 다시 만들 필요 없는 inner 함수를 다음과 같이 생성하자.

useCallback(inner 함수, [...]);

... 부분은 inner 함수 구현에서 캡처한 외부 항목들의 목록이어야 한다.

위와 같이 생성하면 inner 함수는 매번 새로 만들지 않고, 전에 만든 함수를 계속 사용한다.
[...] 부분의 값이 바뀌었을 때만 inner 함수를 새로 만든다.

사실 useCallback 훅을 사용해도 inner 함수는 언제나 새로 만들어진다.
useCallback 훅은 함수이고, 이 함수의 첫째 파라미터가 inner 함수이다.
useCallback 함수가 호출될 때 마다 첫째 파라미터 inner 함수가 생성되어서 파라미터로 전달된다.

그런데 위 코드에서 useCallback 훅은

처음 호출될 때 생성되어 파라미터로 전달된 inner 함수만 보관하고,
그 다음 호출될 때 생성되어 파라미터로 전달된 inner 함수들은 무시한다.
그리고 언제나 처음 전달된 inner 함수만 리턴한다.

과정이야 어쨌든 위 코드의 useCallback 훅이 처음 호출될 때 생성된 하늘색 inner 함수만
계속 사용하게 된다.

앞에서 구현했던 counter 예제를 useCallback을 활용해서 렌더링 최적화를 완성해 보자.

Counter 예제 구현

CounterA.js(전과 동일)

import React from 'react'

function CounterA({ value, increase }) {
	console.log("CounterA 호출됨");
	return (
		<div className="box">
			<h1>CounterA</h1>
			<button onClick={increase}>
				{ value }
			</button>
		</div>
	)
}

export default React.memo(CounterA);

CounterB.js(전과 동일)

import React from 'react'

function CounterB({ value, increase }) {
	console.log("CounterB 호출됨");

CounterContainer.js

import React, { useState, useCallback } from 'react'
import CounterA from './CounterA'
import CounterB from './CounterB'

function CounterContainer() {
	console.log('CounterContainer 호출됨');
	const [countA, setCountA] = useState(0)
	const [countB, setCountB] = useState(0)
	const [value, setValue] = useState(0);

	const increaseA = useCallback(() => { setCountA(countA => countA + 1); }, [setCountA])
	const increaseB = useCallback(() => { setCountB(countB => countB + 1); }, [setCountB])

	return (
		<div className="box">
			<h1>CounterContainer</h1>
			<button onClick={() => setValue(value + 1)}>
				{ value }
			</button>
			<CounterA value={countA} increase={increaseA} />
			<CounterB value={countB} increase={increaseB} />
		</div>
	)
}

export default React.memo(CounterContainer);

increaseA 함수는 setCountA를 캡쳐하였고, countA는 캡쳐하지 않았다.
increaseB 함수는 setCountB를 캡쳐하였고, countB는 캡쳐하지 않았다.

setCountA 함수와 setCountB 함수는 렌더링될 때 마다 새로 만들어지는 inner 함수가 아니고,
렌더링 할 때 마다 항상 같은 함수이다.


useState가 리턴하는 setter 함수는 절대 바뀌지 않고, 언제나 같은 함수를 리턴하기 때문에
useCallback 두 번째 파라미터 배열에 적어줄 필요도 없다.
그래서
  const increaseA = useCallback(() => { setCountA(countA => countA + 1); }, [ ])
  const increaseB = useCallback(() => { setCountB(countB => countB + 1); }, [ ])

따라서,
increaseA, increaseB inner 함수도 렌더링 될 때 마다 새로 만들어지지 않고,
렌더링 할 때 마다 항상 같은 함수이다.
따라서 캡처 목록에 적어주지 않아도 된다.

App.js(전과 동일)

import React from 'react'
import './App.css'
import CounterContainer from './CounterContainer';

function App() {
	console.log('App 호출됨');
	return (
		<div className="box">
			<h1>App</h1>
			<CounterContainer />
		</div>
	);
}

export default React.memo(App);

실행

CounterA 버튼을 클릭하면 CounterContainer와 CounterA 컴포넌트만 렌더링 되고,
CounterB 버튼을 클릭하면 CounterContainer와 CounterB 컴포넌트만 렌더링 되는 것을 확인할 수 있다.

모든 데이터가 CounterContainer의 상태로 구현되었기 때문에,
그 데이터 중 하나가 변경되면 CounterContainer가 렌더링 되어야 하는 것은 피할 수 없다.

따라서 컨테이너 컴포넌트에는

중요한 데이터 상태와, 그 데이터에 대한 작업 함수들만 구현하고,
렌더링 되어야하는 태그 부분은 최소한으로 구현하는 것이 바람직하다.
즉 렌더링 되어야 하는 태그 부분을 프리젠테이션 컴포넌트로 추출해서 구현하는 것이 좋다.
